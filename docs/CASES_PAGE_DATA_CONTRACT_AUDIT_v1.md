# CasesPage Data Contract Audit v1

Scope: `frontend/pages/CasesPage.jsx` audited against `docs/DATA_CONSISTENCY_CONTRACT_v1.md`.

No frontend or backend code was modified.

## Findings

| Status | Location | Contract rule affected | Risk | Suggested fix |
| --- | --- | --- | --- | --- |
| PASS | `loadCasesForEmail()`, approx. lines 905-987 | Truth Source Priority | `/cases?email` records are loaded first, keyed by case id, and same-id local registry records are merged underneath backend records. `hydrateCaseDetails()` then overlays `/case/:caseId` detail data over the merged record. This generally preserves backend precedence for same-named fields. | Keep this merge direction. Add a small helper or comment later to make backend-over-local precedence explicit and testable. |
| WARNING | `loadCasesForEmail()`, approx. lines 929-983 | Truth Source Priority; Page Consumption Contract | `caseRegistry` records are included when their id is absent from `/cases?email`, so purely local cases can appear as active workspace cases. This may be acceptable for drafts, but it also means local snapshots can become visible case records without backend consolidation. | Gate local-only records to draft/new-case display, or label them explicitly as local fallback until backend accepts them. |
| FAIL | `loadCasesForEmail()` hydration mapper, approx. lines 1003-1050 | No Downgrade Rule; Derived Fields Contract | The mapper rewrites `status` to `"active"` when trusted evidence exists, otherwise `c.status || "draft"`. This creates a page-local lifecycle status that can obscure canonical backend statuses such as `receipt_ready`, `receipt_paid`, `verification_ready`, or `verification_issued` if evidence arrays are present. | Preserve canonical backend `status`/`stage` separately and put `"active"` into a display-only field, never back into the case object used by routing/status derivation. |
| WARNING | `deriveCaseListState()`, approx. lines 392-438 and `buildReadinessContract()` dependency | No Downgrade Rule; Derived Fields Contract | `explicitBackendReady` recognizes backend `receiptEligible`, `caseReceiptEligible`, `receiptStatus: ready`, and `stage/status: receipt_ready`, but final `receiptReady` still comes from `buildReadinessContract()`. That helper can return false when evidence is missing, so a backend-ready case can display as not ready if local/hydrated event evidence is absent. | Treat backend receipt-ready signals as monotonic overrides for list display, or update the readiness contract to make explicit backend readiness sufficient unless backend says failed/broken. |
| FAIL | `deriveCaseListState()`, approx. lines 477-490 | No Downgrade Rule; UI status labels | `hasReceiptStageSignal` is assigned only from `hasConcreteReceiptProgress`. The earlier `legacyReceiptReadySignal` detects backend readiness but is not used. A backend `receiptEligible: true` or `receiptStatus: ready` case with `eventCount = 0` and no local events can fall through to `"draft"` or `"Diagnostic completed"` instead of `"Receipt ready"`. | Include canonical backend readiness in `hasReceiptStageSignal`, or use `legacyReceiptReadySignal` as a non-downgrade guard before draft/diagnostic labels. |
| WARNING | `isDiagnosticContinuationCase()`, approx. lines 341-384 | No Downgrade Rule; Stage ordering | The function avoids diagnostic continuation when activated receipt/verification is detected, but it does not directly exclude `receiptEligible`, `caseReceiptEligible`, `receiptStatus: ready`, `stage: receipt_ready`, or `verificationEligible`. A ready backend case with missing event evidence may be classified as diagnostic continuation. | Add explicit guards for receipt-ready and verification-ready backend signals before returning diagnostic continuation. |
| PASS | `deriveCaseListState()`, approx. lines 432-438 and 492-518 | Readiness Contract Usage | Receipt-ready display primarily flows through `buildReadinessContract()` via `readinessContract.receiptReady`, and `"Receipt ready"` is displayed from that contract. | Keep `buildReadinessContract()` as the single readiness display authority, but ensure its inputs honor backend monotonic state. |
| WARNING | `deriveCaseListState()`, approx. lines 392-430 and 477-483 | Readiness Contract Usage | CasesPage precomputes `explicitBackendReady`, `effectiveStructureStatus`, `effectiveStructureScore`, and `hasConcreteReceiptProgress` outside the contract. This means the page partially interprets readiness before the contract runs. | Move these interpretations into the readiness contract or a shared adapter so CasesPage only consumes the contract result. |
| FAIL | `hasReceiptDetailRouteSignal()`, approx. lines 592-631 | Event Trust Boundary; Routing consistency | Raw `eventCount`, `eventCaptured`, `eventLogs`, `events`, `supportingEvents`, and `structuredEvents` can route a case directly to Receipt even without trusted evidence logic. This exceeds the contract allowance for raw event arrays as progress signals only. | Route to Receipt only on backend readiness, activated/payment signals, or `getEvidenceEvents()`/readiness-contract evidence checks, not raw arrays or raw count. |
| WARNING | `hasRealEventSignal()`, approx. lines 668-678 | Event Trust Boundary | Text fields such as `rawEventText`, `eventText`, `captureText`, and `userEventText` count as real event signals without the same case-id and event-type validation used by `getEvidenceEvents()`. They affect post-diagnostic routing and continuation decisions. | Treat raw text as progress context only unless it is attached to a trusted case-scoped event record. |
| WARNING | `deriveCaseListState()`, approx. lines 446-459 and 494-499 | Payment Trust Boundary | `snapshotOnly` blocks `paymentStatus` checkout/paid text from counting as trusted payment progress, but `paid` display still uses `normalized.paymentStatus === "paid"` directly. A snapshot marked `source: "receipt_snapshot"` with `paymentStatus: "paid"` can display `"Paid"`. | Apply the same `!snapshotOnly` guard to the `paid` display calculation, unless a real payment/receipt object exists. |
| WARNING | `hasActivatedReceipt()`, approx. lines 730-758 | Payment Trust Boundary | Activated receipt detection trusts top-level `paid`, `isPaid`, `receiptStatus: paid/activated`, and `paymentStatus`-derived receipt status without checking `snapshotOnly` or payment object provenance. This can route to Verification from a stale snapshot. | Require backend-owned payment/receipt evidence or exclude snapshot-only records from activation checks. |
| PASS | `deriveCaseListState()`, approx. lines 463-468 | Payment Trust Boundary | Receipt object status values `issued`, `activated`, and `paid` are ignored when `source === "receipt_snapshot"`, which prevents those snapshot-only receipt statuses from creating receipt progress. | Preserve this guard and extend the same pattern to all paid/activated checks. |
| WARNING | `hasActivatedVerification()`, approx. lines 761-784 | No Downgrade Rule; Payment/verification trust boundary | Verification activation is detected from verification status values including `ready`, `pass`, `passed`, and `final`, but `verificationEligible` itself is not treated as a canonical display state elsewhere. Verification-ready backend state can be under-displayed, while broad status strings can over-route. | Add a backend-owned verification readiness adapter that distinguishes eligible/ready/paid/issued and excludes snapshot-only signals. |
| PASS | `getCaseDetailRoute()`, approx. lines 634-665 | Routing consistency | Existing cases with receipt or event/receipt progress route to Receipt, activated receipt routes to Verification, missing diagnostic result routes to Diagnostic, and diagnostic result without event signal routes to Pilot/Case Plan (`ROUTES.RESULT`). `caseId` is encoded into every route. | Keep route construction case-id first. Tighten the signal checks as noted above. |
| WARNING | Active list primary action, approx. lines 1626-1632 and 1762-1805 | Routing consistency; No Downgrade Rule | `shouldContinueDiagnostic` depends on `isDiagnosticContinuation`, `!hasReceiptStageSignal`, and `!receiptReady`. Because `hasReceiptStageSignal` ignores backend readiness and `receiptReady` may be false without events, receipt-ready cases can be sent to Pilot/Case Plan instead of Receipt. | Base continuation routing on a monotonic backend-ready guard before checking diagnostic continuation. |
| WARNING | UI status rendering, approx. lines 511-532 and 1691-1699 | UI status labels | Labels are derived from local display logic. A backend `receiptStatus: ready` or `stage: receipt_ready` can show `"Receipt not ready"`, `"Diagnostic completed"`, or raw `draft` if contract evidence is missing. `verificationEligible` is not surfaced as a canonical label. | Derive labels from backend canonical stage/flags first, then contract details as secondary explanatory text. |
| WARNING | Archived list rendering, approx. lines 1848-1887 | UI status labels; Page Consumption Contract | Archived cases reuse the same derived status logic, so any active-list readiness/payment downgrade or over-trust issue also appears in archived cards. | Share a corrected canonical display adapter between active and archived lists. |

## Specific Area Results

| Area | Status | Notes |
| --- | --- | --- |
| Backend truth precedence | WARNING | Initial merge gives backend same-id precedence, but local-only registry records can appear and later hydration rewrites display-bearing fields. |
| No-downgrade rule | FAIL | Backend ready/paid/verification states are not consistently monotonic in display/routing because event absence and diagnostic continuation can override readiness presentation. |
| Readiness contract usage | WARNING | `buildReadinessContract()` is the primary receipt-ready display authority, but CasesPage computes readiness inputs and route/status fallbacks outside the contract. |
| Event trust boundary | FAIL | Raw event arrays/counts can route directly to Receipt via `hasReceiptDetailRouteSignal()`. |
| Payment trust boundary | WARNING | Some snapshot-only payment signals are blocked, but `paymentStatus: paid` and activation checks can still be trusted without provenance checks. |
| Routing consistency | WARNING | Basic route targets and `caseId` propagation are present, but diagnostic continuation can win over backend receipt readiness in edge cases. |
| UI status labels | WARNING | Labels can misrepresent canonical backend state when backend readiness exists but contract evidence is missing locally. |

## Top 3 Risks

| Rank | Risk | Why it matters |
| --- | --- | --- |
| 1 | Backend-ready cases can be displayed or routed as not ready/diagnostic continuation when event evidence is missing locally. | Violates the no-downgrade rule for `receipt_ready` and can send users to the wrong workflow. |
| 2 | Raw event arrays and counts can route cases to Receipt without trusted evidence logic. | Violates the event trust boundary and can overstate receipt readiness. |
| 3 | Snapshot-only or weak payment signals can still create `"Paid"` display or activation routing in some paths. | Violates payment source ownership and can overstate paid/verification state. |
