# STEP 5.17 Event Persistence Audit

Date: 2026-05-08

Scope: event capture and event persistence behavior in the current Nimclea frontend/backend codebase. This is documentation only; no application logic was changed.

## Executive Summary

`ReceiptPage` Quick Capture is the active production event capture path. The active submit handler, `handleQuickCaptureSubmit`, writes a local case update and then calls `logTrialEvent(...)`.

`logTrialEvent` in `frontend/lib/trialApi.js` writes to `POST /event/log`. The backend `POST /event/log` route persists a normalized event record into `backend/data/eventLogs.json` and mirrors that event to Supabase `event_logs` when Supabase is configured.

`EventCaptureBox.jsx` exists in the codebase, but current frontend search results do not show it imported or rendered anywhere. Its submit path uses `caseRegistry.addCaseEvent(caseId, eventPayload)`, which writes only to `localStorage`. It should be treated as legacy/dormant and must not be used as a formal evidence capture path unless it is wired to `/event/log`.

## Active Durable Event Path

| Step | Code path | Persistence result |
|---|---|---|
| User submits Quick Capture | `frontend/pages/ReceiptPage.jsx` `handleQuickCaptureSubmit` | Builds `quickCaptureEvent` with `caseId`, type, signal impact, note, and capture time. |
| Local case cache update | `upsertCase({ caseId, events: [...] })` | Updates frontend `localStorage` case registry/cache. |
| Durable event write | `logTrialEvent({ eventType: "quick_capture_submitted", page: "ReceiptPage", ... })` | Calls the shared durable event API wrapper. |
| API wrapper | `frontend/lib/trialApi.js` `logTrialEvent` | Sends `POST ${API_BASE}/event/log` with enriched `caseId` and `meta.caseId`. |
| Backend route | `backend/routes/eventRoutes.js` `router.post("/log")` | Validates `eventType`, creates server `eventId`, appends to `eventLogs.json`. |
| Mirror side effect | `backend/utils/fileStore.js` `appendJsonFile` | When basename is `eventLogs.json`, calls `mirrorEventLogToSupabase(item)`. |
| Supabase mirror | `backend/utils/supabaseMirrorWrites.js` `mirrorEventLogToSupabase` | Upserts to Supabase `event_logs` by `event_id` when Supabase is enabled. |

Durable storage target:

- `backend/data/eventLogs.json`
- Supabase `event_logs` mirror, when configured

Primary keys/identifiers:

- `eventId`, generated by backend
- `caseId`
- `userId`
- `trialId`
- `sessionId` in frontend payload/meta
- `meta.quickCaptureId` for the original quick capture id

## Dormant Local-Only Path

`frontend/components/EventCaptureBox.jsx` is present but was not found imported or rendered anywhere in the current frontend search results.

Its capture path:

1. Builds an `eventPayload` in `EventCaptureBox.jsx`.
2. Calls `addCaseEvent(caseId, eventPayload)`.
3. `addCaseEvent` in `frontend/utils/caseRegistry.js` writes to `localStorage` key `nimclea_case_${caseId}`.
4. `addCaseEvent` then calls `upsertCase(nextCase)`, which updates `localStorage` key `nimclea_case_registry` and `nimclea_current_case_id`.

This path does not call `logTrialEvent`, `POST /event/log`, or any backend route. It does not write `eventLogs.json` and does not mirror to Supabase.

Classification:

- `EventCaptureBox.jsx`: legacy/dormant local-only capture component.
- `caseRegistry.addCaseEvent`: local cache helper, not a durable evidence writer.

## Risk Assessment

1. If `EventCaptureBox.jsx` is reintroduced as an evidence capture UI without backend wiring, users could believe events were durably captured while the evidence exists only in browser `localStorage`.
2. Local-only events can be lost when browser storage is cleared, when a different browser/device is used, or when local case cache is overwritten.
3. Backend reads that depend on `eventLogs.json`, including case hydration and analytics routes, will not see events captured only through `addCaseEvent`.
4. Supabase `event_logs` will not receive local-only `addCaseEvent` events, creating a gap between formal event evidence and UI-visible local event history.
5. The active `ReceiptPage` path currently performs both a local cache write and a durable event log write. If the `logTrialEvent` request fails, the local cache can still show the event while durable evidence persistence fails.

## Recommendation

- Treat `ReceiptPage` Quick Capture as the formal production event capture path.
- Treat `EventCaptureBox.jsx` as dormant legacy code unless it is explicitly wired to `logTrialEvent` or directly to `POST /event/log`.
- Keep `caseRegistry.addCaseEvent` scoped as a local cache helper only.
- For any future evidence capture UI, require durable persistence through `/event/log` before representing the event as formal evidence.
- Consider adding a small code comment or documentation reference near `EventCaptureBox.jsx` and `caseRegistry.addCaseEvent` in a future cleanup so the local-only behavior is clear to future implementers.

## No Code Changes Made

No application logic, route behavior, or frontend behavior was modified for this audit. The only change is this markdown document.
